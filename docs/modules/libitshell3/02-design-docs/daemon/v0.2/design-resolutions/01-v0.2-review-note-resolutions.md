## Resolution 1: 16-Pane Limit and Fixed-Size Data Structures (6/6 unanimous)

**Source**: Review note 01 (HIGH)
**Affected docs**: 01-internal-architecture.md (Sections 1.5, 1.6, 3.2, 3.3), design-resolutions/01-daemon-architecture.md (R1, R3, R9)

### Decision

Adopt a 16-pane-per-session limit. Replace all dynamic data structures in the session state tree with fixed-size alternatives: array-based binary tree, fixed pane slot arrays, dirty bitmap, and fixed ring cursor arrays.

### 1.1 MAX_PANES_PER_SESSION = 16

A compile-time constant in `core/`:

```zig
pub const MAX_PANES = 16;
pub const MAX_TREE_NODES = MAX_PANES * 2 - 1; // 31
```

The 16-pane limit is a UX-driven constraint, not a performance optimization. On a 374x74 terminal, 16 panes at 93x18 is minimum viable; 32 panes at 93x9 is unusable. The limit is enforced server-side via ErrorResponse when a client requests a split that would exceed 16 panes. The client does not receive limits during handshake — the server is the single source of truth for resource limits.

### 1.2 Array-Based Binary Tree

Replace the pointer-based `SplitNode` tree with an inline array:

```zig
// core/split_node.zig
pub const SplitNodeData = union(enum) {
    leaf: PaneSlot,
    split: struct {
        orientation: enum { horizontal, vertical },
        ratio: f32,
    },
};
```

The `Session` struct embeds the tree inline:

```zig
// core/session.zig
pub const Session = struct {
    // ... existing fields ...
    tree_nodes: [MAX_TREE_NODES]?SplitNodeData, // 31 entries
    // root is always tree_nodes[0]
};
```

Navigation uses index arithmetic: `parent(i) = (i-1)/2`, `left(i) = 2*i+1`, `right(i) = 2*i+2`. No child pointers, no per-node heap allocation.

**Tree node array vs pane slot array**: These are separate index spaces. Tree node indices (0..30) identify positions in the `[31]?SplitNodeData` array. Pane slot indices (0..15) identify positions in the `[16]?*Pane` array. Leaf nodes store pane slot indices. Tree compaction (subtree relocation during split/close) moves tree nodes but does not change the pane slot indices stored in leaf values. Pane slot indices are stable across tree mutations.

**Tree mutation complexity**: Split and close operations require subtree relocation within the tree node array. With max depth 4 and 31 nodes, this is bounded at ~15 node copies per operation — trivially fast on cache-hot data (the entire tree fits in L1 cache).

**Exact byte-level layout**: Deferred to implementation. The architectural invariants are: 31 nodes, inline in Session, no heap allocation. Whether Zig packs `?SplitNodeData` into 8 or 16 bytes does not affect the architecture.

### 1.3 Fixed-Size Pane Lookup

Replace `HashMap(PaneId, *Pane)` with a per-session fixed array:

```zig
pane_slots: [MAX_PANES]?*Pane, // indexed by PaneSlot (0..15)
```

Each session maintains its own pane slot array. `SessionManager` continues to use `HashMap(u32, *Session)` for sessions (dynamic count, few instances — no fixed limit proposed for sessions).

**Free slot tracking**: A `free_mask: u16` bitmap tracks available slots. `@ctz(free_mask)` gives the next free slot in one instruction.

### 1.4 PaneId Semantics: Global Monotonic u32 (Sub-Resolution)

**Decision**: PaneId on the wire remains a global monotonic `u32`, never reused within daemon lifetime. Internally, sessions use a session-local slot index (`PaneSlot: u8`, 0..15) for all fixed-size array operations.

**Wire**: `pane_id: u32` in JSON payloads. Globally unique, server-assigned from a monotonic counter (`next_pane_id: u32`). Never reused — closed pane IDs are permanently retired.

**Internal**: `PaneSlot: u8` (0..15). Used for all hot-path operations: dirty bitmap, pane slot array indexing, ring cursor array indexing, tree leaf values.

**Mapping**: Wire-to-Pane lookup via per-session linear scan of `pane_slots` (at most 16 entries). This runs on the cold path only (protocol message dispatch at the `server/` boundary). Hot paths (frame export, dirty iteration, PTY read) use slot indices exclusively and never touch PaneId.

**`Pane` struct fields**:
- `pane_id: u32` — global wire identity
- `slot_index: u8` — position in the owning session's `pane_slots` array, for O(1) dirty mask operations (`dirty_mask |= (1 << pane.slot_index)`)

**Rationale — why not session-local PaneId (0..15) on the wire**:

Three arguments drove unanimous convergence on global monotonic PaneId:

1. **Pane-reuse race condition**: In async IPC, a client can have an in-flight message targeting a pane slot that the server has already destroyed and recycled for a new pane. With session-local PaneId = slot index, the stale message targets the wrong pane. With global monotonic PaneId, the stale message targets a non-existent ID and receives an ErrorResponse. This is a real race — the client sends asynchronously, and messages can cross in flight with server-side state changes.

2. **Protocol constraint leak**: Session-local PaneId (0..15) exposes the 16-pane limit to wire protocol semantics. Clients would implicitly know pane_id is bounded to 0..15, leaking a server-side implementation detail. Global monotonic u32 keeps the limit invisible to the protocol.

3. **Hot-path equivalence**: Both options have identical hot-path performance. All hot paths (dirty bitmap iteration, frame export, focused pane lookup, PTY read handling) use session-local slot indices, never PaneId. The difference is only at the protocol boundary (cold path), where the linear scan of 16 entries is negligible.

**Retracted argument**: An earlier argument that subtree compaction would change pane identifiers under session-local PaneId was retracted. Tree node indices (0..30) and pane slot indices (0..15) are separate index spaces; compaction does not affect pane slot indices. This retraction did not affect the conclusion — the race condition and constraint-leak arguments are independently sufficient.

### 1.5 Dirty Bitmap

Add a per-session dirty tracking bitmap in `server/`:

```zig
dirty_mask: u16, // one bit per pane slot
```

Set via `dirty_mask |= (1 << pane.slot_index)` on PTY read. Iterate via `@ctz(dirty_mask)` (single instruction on x86 `TZCNT` and ARM64 `RBIT+CLZ`). Clear each bit after export: `dirty_mask &= dirty_mask - 1`.

### 1.6 Fixed-Size Ring Cursors

Replace `HashMap(PaneId, RingCursor)` in `ClientState` with:

```zig
ring_cursors: [MAX_PANES]?RingCursor, // indexed by PaneSlot (0..15)
```

Same indexing scheme as the pane slot array. Indexed by session-local slot, not by PaneId.

### 1.7 Impact on Existing Resolutions

| Resolution | Change |
|-----------|--------|
| R1 (Module Decomposition) | Section 1.5: pane lookup changes from `HashMap(PaneId, *Pane)` to `[16]?*Pane`. |
| R3 (State Tree) | `SplitNode` type replaced by `SplitNodeData`. Session struct gains `tree_nodes: [31]?SplitNodeData`. `root: ?*SplitNode` removed. Hierarchy diagram updated. |
| R9 (Client Connection) | `ClientState.ring_cursors` changes from `HashMap(PaneId, RingCursor)` to `[16]?RingCursor`. |

### 1.8 Wire Protocol Changes

**None.** The 16-pane limit, array-based tree, and fixed-size arrays are purely internal optimizations. PaneId on the wire stays `u32`. No new message types, no modified payloads, no capability changes.

## Resolution 2: IME Module Naming (6/6 unanimous)

**Source**: Review note 02 (MEDIUM)
**Affected docs**: 01-internal-architecture.md (Sections 1.1, 1.2, 1.3, 1.6), design-resolutions/01-daemon-architecture.md (R1)

### Decision

Two changes:

### 2.1 Add `libitshell3-ime` to Inter-Library Dependency Diagram

`libitshell3-ime` is a project-level library at the same level as `libitshell3-protocol`. It was missing from the diagram in Section 1.6 of doc 01. The updated diagram:

```
libitshell3-protocol  (standalone — depends only on Zig std; libssh2 added in Phase 5)
libitshell3-ime       (standalone — depends on libhangul)
libitshell3/core/     (standalone — no external deps)
libitshell3/ghostty/  (depends on core/, vendored ghostty)
libitshell3/input/    (depends on core/)
libitshell3/server/   (depends on core/, ghostty/, input/, libitshell3-ime, libitshell3-protocol)
```

### 2.2 Rename Internal `ime/` Module to `input/`

The internal module `libitshell3/ime/` is renamed to `libitshell3/input/` to disambiguate from the external library `libitshell3-ime` (HangulImeEngine).

**Scope statement**: The `input/` module handles Phase 0+1 key input processing: shortcut interception (Phase 0), ImeEngine dispatch (Phase 1), focus change handling (`handleIntraSessionFocusChange`), and input method switching (`handleInputMethodSwitch`). Mouse events and paste operations bypass this module entirely — they are handled directly in `server/`.

The dependency diamond updates:

```
          core/
         /    \
    ghostty/  input/
         \    /
         server/
```

**Rationale**:

Candidates considered:
- `input/` — chosen. Short, describes the domain (input processing that requires composition/routing decisions). Clean disambiguation from `libitshell3-ime`. No "ime" in the name eliminates the naming collision entirely.
- `key_routing/` — rejected. Too specific. The module also handles focus change events and input method switching, which are not "key routing."
- `ime_routing/` — rejected. Still contains "ime," partially defeating the disambiguation goal.

**Note on ghostty precedent**: ghostty's `input/` module includes mouse, paste, key, bindings — the full input scope. Our `input/` module is narrower (key input only). The name `input/` is broader than the current scope, which is why the scope statement above is important for documentation clarity.

### 2.3 Wire Protocol Changes

**None.** This is an internal module rename with no protocol impact.

## Resolution 3: Protocol Library Scope Misstatement (6/6 unanimous)

**Source**: Review note 03 (MEDIUM)
**Affected docs**: 01-internal-architecture.md (Section 1.4)

### Decision

Replace the incorrect scope statement in Section 1.4 (Ring Buffer Placement):

**Current (incorrect)**:
> "The protocol library defines message formats, not delivery mechanisms. There is no client-side analogue."

**Replacement (correct)**:
> "The ring buffer is a server-side application-level delivery optimization (multi-client cursor management, writev scheduling) with no client-side analogue. The protocol library provides transport-level I/O (Layer 4), but application-level delivery strategies are the consumer's responsibility."

**Rationale**: The current text reduces the protocol library's scope to Layer 1 (message formats) only. Per Resolution 5 of the v0.1 design resolutions, the protocol library owns four layers: L1 Codec, L2 Framing, L3 Connection Protocol, and L4 Transport (socket lifecycle, Listener, Connection). Layer 4 IS a delivery mechanism. The corrected text accurately distinguishes between transport-level I/O (protocol library, Layer 4) and application-level delivery strategies (server's ring buffer in `server/`).

### 3.1 Wire Protocol Changes

**None.** This is a documentation correction only.

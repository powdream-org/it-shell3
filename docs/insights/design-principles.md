# Design Principles

Living document. Updated after each revision cycle as principles emerge or are
reinforced. For the full context behind each principle, see the originating
handover document.

## Protocol Design Principles

| #  | Principle                                            | Origin                 | Notes                                                                                                                                                                                                                                                                                                                                                                                   |
| -- | ---------------------------------------------------- | ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| P1 | **Shared state by default**                          | v0.7, reinforced v0.10 | All clients see the same viewport, selection, scroll position. Per-client divergence requires explicit justification. Heuristic: "does tmux do this per-client?" If no, neither do we. v0.10 R1 reverted scroll I-frame from per-client direct queue back to shared ring buffer.                                                                                                        |
| P2 | **One delivery path**                                | v0.7, reinforced v0.8  | Ring buffer is the canonical data path. The direct message queue (for small control messages like PreeditSync, LayoutChanged) is the sole exception. New features route through the ring unless strongly justified otherwise.                                                                                                                                                           |
| P3 | **Close early, design later**                        | v0.7                   | Open questions with no concrete use case should be closed, not carried forward. They can be reopened when a real scenario demands it.                                                                                                                                                                                                                                                   |
| P4 | **Implementation details are not protocol concerns** | v0.7, reinforced v0.11 | Clipboard policy, cursor blink timing, focus indicator rendering — these belong to the client app, not the wire protocol. v0.11 extended this to daemon behavior: coalescing tiers, health timelines, ring buffer architecture, resize policies, preedit ownership algorithms all moved from protocol docs to daemon docs. Protocol docs retain wire format and message semantics only. |
| P5 | **ghostty is the rendering authority**               | v0.8                   | Cursor positioning, preedit cell width, decoration style — all determined by ghostty's renderer on the server side. The protocol carries cell data, not rendering instructions. If ghostty handles it internally, the protocol should not duplicate it.                                                                                                                                 |
| P6 | **Capability gating is precise**                     | v0.8                   | Each capability gates exactly one thing. `preedit_sync` gates only PreeditSync (0x0403). `preedit` gates PreeditStart/Update/End. Preedit cell data in FrameUpdate is never gated. Layers are independent.                                                                                                                                                                              |

## Architectural Insights

| #   | Insight                                                                      | Origin                 | Impact                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| --- | ---------------------------------------------------------------------------- | ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| A1  | **Preedit is cell data, not metadata**                                       | v0.7 PoC, applied v0.8 | Server calls `ghostty_surface_preedit()`, injects preedit cells into I/P-frames. Client renders cells without knowing what is preedit. Eliminated dual-channel design, `composition_state`, FrameUpdate preedit JSON, ring buffer bypass.                                                                                                                                                                                                     |
| A2  | **Globally singleton session model**                                         | v0.7                   | All clients share the same session state. No per-client independent viewports or scroll positions (post-v1 concern).                                                                                                                                                                                                                                                                                                                          |
| A3  | **Ring buffer resolves adjacent problems**                                   | v0.7                   | Flow control (cursor stagnation replaces explicit ack), notification coalescing (per-pane ring makes batching unnecessary), recovery unification (skip to latest I-frame), selection sync (shared state in RowData). Consider the ring buffer first before inventing alternatives.                                                                                                                                                            |
| A4  | **Client is a thin RenderState populator, not a custom renderer**            | PoC 08 (GPU verified)  | `importFlatCells()` populates RenderState directly → ghostty's existing `rebuildCells()` + `drawFrame()` handles all rendering. No Terminal, no VT parser, no manual font resolution or GPU buffer construction on the client. See [ghostty-api-extensions.md](ghostty-api-extensions.md).                                                                                                                                                    |
| A5  | **CellData is semantic, not GPU-ready**                                      | PoC 07–08              | Wire format carries semantic content (codepoint, style, color) for RenderState population. GPU structs (CellText, CellBg) are 70%+ client-local data built by `rebuildCells()`. Export+import = 34 µs for 80×24 (0.2% of frame budget).                                                                                                                                                                                                       |
| A6  | **FlatCell is a flattened Style-resolved projection**                        | v0.9                   | ghostty's `page.Cell` (8B) stores `style_id` → `RefCountedSet` lookup → `Style`. Wire FlatCell (16B) resolves this indirection and inlines colors/flags. Server export cost is dominated by style dereference (~11 ns/cell). Client import skips indirection entirely (~4 ns/cell). The wire format is NOT a memcpy of ghostty's Cell — it's a denormalized transformation.                                                                   |
| A7  | **SSH compression makes application-layer compression unnecessary for v1**   | v0.9                   | SSH's `zlib@openssh.com` (persistent dictionary, transport-layer) achieves 10-20x on typical FlatCell data (empty cells = zeros, repeated colors). No application-layer compression in v1. The `COMPRESSED` header flag is reserved but unused. Only matters for remote (SSH tunnel) — local Unix socket bandwidth is not a constraint (>1 GB/s).                                                                                             |
| A8  | **RenderState dirty flag is the single source of truth for frame decisions** | v0.10                  | Every terminal state change (PTY output, cursor, modes, colors, preedit) flows through ghostty's RenderState. No bypass paths exist — verified across protocol docs and IME contract implementation. Frame type selection (P-frame vs I-frame) reduces to one boolean per pane. Don't maintain parallel tracking (byte comparison, generation counters) when the engine already tracks mutations.                                             |
| A9  | **Static allocation cascades into type design**                              | v1.0-r8 (impl)         | ADR 00052 (static SessionManager) + ADR 00058 (fixed-size inline buffers) transformed Session fields from `[]const u8` slices to `[N]u8` + `_length` pairs, SessionManager from HashMap to fixed array. This eliminated allocator dependencies from all core types, enabled `.bss` allocation, and simplified test isolation. Pre-implementation type designs that assume dynamic allocation may not survive contact with Zig's memory model. |
| A10 | **Protocol is serialization; transport is OS**                               | v1.0-r8 (impl)         | ADR 00060 split libitshell3-protocol into protocol (codec, framing, wire format) and transport (SocketConnection, Listener). Connection state machine moved to daemon library. Polymorphic vtable eliminated. When a library mixes "how to encode" with "how to send," the boundary is wrong.                                                                                                                                                 |

## Process Lessons

| #   | Lesson                                                                        | Origin                      | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| --- | ----------------------------------------------------------------------------- | --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| L1  | **Normative table rows must be self-contained**                               | v0.8 (verification cascade) | When rows in a table describe variants of the same mechanism, each row must reference the authoritative section by number rather than paraphrasing its rules. Fixing one row can expose adjacent rows as inconsistent. Check all rows when any single row changes.                                                                                                                                                                               |
| L2  | **Stop verification at diminishing returns**                                  | v0.8 (11 rounds)            | Stop when: (a) issues have declined to minor/marginal severity, (b) new issues are in the same file/section as the previous round's fix, and (c) at least 3/4 verifiers report CLEAN.                                                                                                                                                                                                                                                            |
| L3  | **Single-topic revision cycles converge faster**                              | v0.8 vs v0.7                | v0.8 (1 topic, 16 resolutions, 7/7 unanimous) converged faster than v0.7 (2 topics, more complex coordination). Scope discipline accelerates consensus.                                                                                                                                                                                                                                                                                          |
| L4  | **Mechanical fixes converge; prose fixes cascade**                            | v0.9, reinforced v1.0-r9    | v0.9 verification: Round 1 (5 issues) → Round 2 (4, including 1 regression from prose fix introducing phantom message name) → Round 3 (3, including 1 minor regression from precedence ambiguity). Regressions came from prose changes (new sentences with references), never from arithmetic/number fixes. When fixing verification issues, prefer mechanical substitutions over new prose.                                                     |
| L5  | **Verify assumptions at module boundaries**                                   | v0.10                       | The "dirty flag is sufficient" claim required cross-module verification (protocol × IME contract) to confirm no bypass paths. Single-doc analysis is insufficient for cross-cutting claims. When a design decision depends on behavior in another module, verify at the boundary.                                                                                                                                                                |
| L6  | **Cross-module content migration is rewriting, not copying**                  | v0.3 (daemon)               | Absorbing content from IME/protocol docs into daemon docs introduced 3 headless API violations (V1-01, V1-02, R2-03) — Surface API calls that contradict the daemon's headless architecture. Source text must be _translated_ to the target module's architecture, not moved verbatim.                                                                                                                                                           |
| L7  | **Stubbing a section requires cross-reference grep**                          | v0.8 (IME)                  | Converting IME v0.8 §5 to a reference stub left stale cross-references in daemon docs 01 and 02 (R2-06, R2-07). When replacing a section with a stub, grep all modules for citations pointing to that section.                                                                                                                                                                                                                                   |
| L8  | **Enumerating items in a fix note requires per-item verification**            | IME behavior v1.0-r1        | A fix note that lists multiple items under a single behavioral claim ("all of these → path X") will be wrong for any exception in the group. The Round 3 fix for isPrintablePosition() correctly identified 5 non-printable HID codes but incorrectly grouped all 5 under "flush/forward path" — Backspace belongs to a distinct third path (IME undo). Verify each enumerated item independently against all other documents before committing. |
| L9  | **Summary/aggregation sections are divergence vectors**                       | protocol v1.0-r12           | A section that restates behavior already defined in individual message sections (e.g., "Multi-Client Behavior" assembling per-message broadcast rules) diverges from the originals over time. The fix is deletion: individual message definitions are the authoritative record. Cross-reference where summary is genuinely useful.                                                                                                               |
| L10 | **Convention compliance is structural, not cosmetic**                         | daemon v1.0-r5              | Metadata conventions (Date/Scope only) and cross-doc reference rules (no exact paths to independent revision cycles) prevent structural breakage, not just style drift. Exact-path links break on every revision; changelog metadata in headers obscures the two fields implementers need. Retroactive enforcement found violations in all 4 daemon spec docs — treat convention checks as part of review, not optional polish.                  |
| L11 | **Cross-doc fixes require single-writer issue clusters**                      | daemon v1.0-r6              | When a fix spans multiple documents, parallel writers make independent judgment calls that diverge. Assigning one writer per doc (the initial-writing pattern) does not work for fix rounds. Group fixes by issue cluster — one writer per cluster, even across docs. Mandatory same-class sweep before spawning fix writers to identify all related locations.                                                                                  |
| L12 | **Intra-procedure step reordering requires resource-dependency trace**        | daemon v1.0-r7              | When a fix defers or reorders a step within a procedure, cascade analysis must trace which resources the moved step consumes and whether any intervening step invalidates them. V2-01 deferred `deactivate()` past `close(pty_fd)` — use-after-close caught only in Round 3. Cross-document cascade analysis alone is insufficient; intra-procedure ordering must be checked explicitly.                                                         |
| L13 | **Normative tables referencing code paths require bidirectional propagation** | daemon v1.0-r7              | When a resolution adds a normative table (e.g., cleanup triggers) that references existing code paths, those code paths must be updated in the same writing round. CTR-13 defined 6 cleanup triggers in doc04 §11.6 but 4 referenced code paths (doc03 §3.3, §4.2; doc04 §8.2) were not updated by the initial writers. Verification should not be the first place these gaps are discovered.                                                    |
| L14 | **Implementation validates specs more thoroughly than review alone**          | daemon v1.0-r8 (impl)       | Plans 1-7.5 implementing from v1.0-r8 specs produced 17 ADRs and 9 CTRs — far more design refinement than the 3-round verification cycle found. Writing code forces resolution of ambiguities that review misses (buffer strategies, type representations, module boundaries). When a spec is mature enough to implement, implementation feedback should flow back before the next design consumers depend on it.                                |
| L15 | **Numeric constant changes cascade through ALL representation forms**         | Plan 15 v1.0-r9 (unified)   | Header size 16→20 took 4 verification rounds because "16" appeared as prose text (Round 1), bare number in offset tables and performance calculations (Round 2), and hex dump byte layout (Round 3). Same-class sweep must cover ALL forms: prose, tables, pseudocode, offset tables, arithmetic, hex dumps, Mermaid diagrams. Grepping for "16-byte" misses bare "16" in `Header: 16 bytes` and `0 \| 16 \| [header]`.                          |

## Maintenance Rules

1. **When to update**: After completing a revision cycle's handover document.
   Review §2 (Insights) and §3 (Design Philosophy) of the new handover — if a
   principle is new, add it; if it reinforces an existing one, update the Origin
   column.
2. **Supersession**: If a later revision overturns a principle, do not delete
   it. Move it to a "Retired" section with a note explaining why and which
   principle replaced it.
3. **Scope**: Only principles that have been **validated through at least one
   revision cycle** belong here. Speculative ideas stay in handover documents
   until confirmed.

# PoC 08: RenderState Re-injection

**Date**: 2026-03-08
**Triggered by**: PoC 07 (bulk export validated) — now proving the full round-trip: export → transfer → re-inject → render
**Question**: Can FlatCell data extracted from one ghostty Terminal be re-injected into a completely separate Terminal instance, producing identical RenderState output?

## Motivation

PoC 07 proved fast bulk export of RenderState into FlatCell[]. But for the it-shell3 protocol, the client must:

1. **Receive** FlatCell data from the server over the wire
2. **Re-inject** it into a local ghostty Terminal instance
3. **Render** via the normal GenericRenderer → Metal GPU pipeline

This PoC validates step 2: feeding extracted cell data back into a fresh Terminal and verifying the output matches the original.

## What Was Built

### Re-injection algorithm

Uses ghostty's high-level Terminal API (`setCursorPos` + `setAttribute` + `print`):

```zig
fn reinject(term: *Terminal, cells: [*]FlatCell, rows: u16, cols: u16) !void {
    for (0..rows) |y| {
        for (0..cols) |x| {
            const cell = cells[y * cols + x];

            // Skip spacer cells — auto-generated by print() for wide chars
            if (cell.wide == 2 or cell.wide == 3) continue;
            if (cell.codepoint == 0) continue;

            // Position cursor
            term.setCursorPos(y + 1, x + 1);  // 1-indexed

            // Set style (only when changed)
            term.setAttribute(.unset);
            applyStyle(term, cell);  // decode flags, fg, bg

            // Print — handles wide char mechanics automatically
            term.print(cell.codepoint);
        }
    }
}
```

### Style reconstruction

FlatCell.flags (u16) maps directly to ghostty's `Style.Flags` packed struct:

| Bit | Attribute |
|-----|-----------|
| 0 | bold |
| 1 | italic |
| 2 | faint |
| 3 | blink |
| 4 | inverse |
| 5 | invisible |
| 6 | strikethrough |
| 7 | overline |
| 8-10 | underline (3-bit enum: none/single/double/curly/dotted/dashed) |

Colors are reconstructed from PackedColor:
- tag=0 → none (skip)
- tag=1 → palette index (via `@"256_fg"` / `@"256_bg"`)
- tag=2 → direct RGB (via `direct_color_fg` / `direct_color_bg`)

### Optimizations

- **Style change tracking**: Only calls `setAttribute` when fg/bg/flags differ from previous cell
- **Skip spacer cells**: `print()` auto-generates spacer_tail for wide characters — the re-injector skips them
- **Skip empty cells**: Cells with codepoint=0 are left at Terminal's default (empty)

### Analysis

- **Re-injection is faster than export**: 39 µs vs 119 µs for 80×24. This is because re-injection only processes non-empty cells (106 out of 1920), while export processes all cells.
- **Re-injection throughput**: ~2.7 µs per non-empty cell (39 µs / ~14 cells per row × ~rows-with-content). The per-cell cost is dominated by setAttribute + print API calls.
- **Full round-trip at 60 fps**: 158 µs for 80×24 = **0.95%** of 16.6 ms frame budget. Even 200×50 uses only **3.8%**.

## Findings

### 1. High-level API works perfectly for re-injection

`Terminal.print()` + `setAttribute()` + `setCursorPos()` is the correct approach:
- `print()` automatically handles wide character mechanics (`.wide` flag + `.spacer_tail` generation)
- `setAttribute()` correctly manages the page-local StyleSet (ref-counted style deduplication)
- No need for low-level page.Cell manipulation

### 2. Style_id is page-local — semantic style must be on the wire

`page.Cell.style_id` is an index into a page-specific `RefCountedSet`. It is meaningless outside the page where it was created. The wire format **must** carry resolved semantic style (fg_color, bg_color, flags), not style_id. The receiver calls `setAttribute()` which internally resolves to a local style_id.

### 3. Spacer cells should NOT be transmitted

`spacer_tail` and `spacer_head` cells are implementation artifacts. The sender should skip them; the receiver's `print()` generates them automatically from the wide flag. This saves ~1 byte per wide character column.

### 4. Palette vs RGB distinction matters

ghostty preserves whether a color was set via palette index or direct RGB. If the user's color palette changes at runtime, palette-indexed colors update but RGB colors don't. The wire protocol (FlatCell.PackedColor) correctly preserves this via the tag field.

### 5. Empty cells can be skipped

For a typical terminal screen, most cells are empty (1814 out of 1920 in the test). Re-injection only needs to process non-empty cells. Combined with dirty-row tracking from PoC 07, the typical data volume is very small.

### 6. Round-trip is well within real-time budget

The full round-trip (export + reinject + re-export) for 80×24 is 158 µs, under 1% of a 60 fps frame budget. The protocol overhead (serialization, network, deserialization) will add more, but the computational core is fast.

## Impact on Protocol Design

### Wire format validation

FlatCell (16 bytes) carries exactly the right information:
- `codepoint` (u32) — the character
- `fg`, `bg` (PackedColor, 4B each) — tag + RGB or palette index
- `flags` (u16) — style flags, bitwise compatible with ghostty internals
- `wide` (u8) — needed to skip spacer cells, but spacer cells themselves are NOT sent
- `content_tag` (u8) — for future grapheme/bg-color-only handling

### Client architecture confirmed

```
Server: Terminal → RenderState.update() → bulkExport() → FlatCell[] → wire
Client: wire → FlatCell[] → reinject() → Terminal → RenderState.update() → rebuildCells() → GPU
```

The client creates its own Terminal instance and populates it via the re-injection algorithm. Then the normal ghostty rendering pipeline (RenderState → GenericRenderer → Metal) takes over.

### Bandwidth estimation (updated)

Since only non-empty cells need transmission:

| Scenario | Non-empty cells | Wire data | At 60 fps |
|----------|----------------|-----------|-----------|
| 80×24 typical (2 rows active) | ~160 | 2.5 KB | 150 KB/s |
| 80×24 full screen | ~1,920 | 30 KB | 1.8 MB/s |
| 200×50 typical (5 rows active) | ~1,000 | 16 KB | 960 KB/s |
| 200×50 full screen | ~10,000 | 160 KB | 9.6 MB/s |

With dirty-row filtering + empty-cell skipping, typical throughput is well under 1 MB/s.

## Known Limitations

- **No grapheme cluster support**: Only the primary codepoint is transferred. Multi-codepoint graphemes (content_tag=1) would need the additional codepoints from `RenderState.Cell.grapheme`.
- **bg-color-only cells**: Cells with `content_tag` of `bg_color_palette`/`bg_color_rgb` have no codepoint. These need special handling (set bg color + print space, or direct cell manipulation).
- **Row flags not transferred**: `page.Row` flags like `wrap`, `semantic_prompt` are not part of FlatCell. For exact reproduction of wrap behavior, these would need separate handling.
- **Cursor position not restored**: The PoC does not restore the cursor to Terminal A's position. For the protocol, cursor position is transmitted in ExportResult.
- **Performance not yet production-optimized**: Terminal.init/deinit per frame (in benchmark) is unrealistic. Production would reuse the Terminal and use `clear` + reinject.

## Next Steps

1. **Buffer reuse**: Reuse Terminal B across frames instead of init/deinit each time
2. **Delta injection**: Only reinject dirty rows (using dirty_bitmap from ExportResult)
3. **Grapheme support**: Handle multi-codepoint cells via Terminal's grapheme clustering mode
4. **Wire serialization**: Add LZ4/zstd compression on the FlatCell stream
5. **GPU rendering**: Connect re-injected Terminal to GenericRenderer → Metal (requires Surface, not headless)
